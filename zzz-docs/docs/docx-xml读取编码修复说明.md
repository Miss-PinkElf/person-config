# docx XML 读取编码修复说明

更新时间：2026-04-03

## 背景

在 `.codex/skills/docx/scripts/office/validators/base.py` 中，校验 Office 文档内部 XML 文件时，原先使用的是文本模式读取：

```python
with open(xml_file, "r") as f:
    xml_doc = lxml.etree.parse(f)
```

这在部分环境下可以工作，但在 Windows 环境中存在明显隐患。

## 为什么要修复

### 1. XML 文件本身可能声明了自己的编码

Office 文档内部的 XML 文件通常会带有编码声明，例如：

```xml
<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
```

这意味着解析器应该依据 XML 文档自身声明的编码去解析内容，而不是依赖操作系统默认编码。

### 2. 文本模式读取会先经过 Python 的默认解码

如果使用 `open(xml_file, "r")`，Python 会先按当前环境默认文本编码读取文件内容。

在 Windows 下，这个默认编码往往不是 XML 实际使用的编码。这样会带来两个问题：

- 还没交给 `lxml` 之前，文件内容就已经被错误解码
- XML 中声明的编码信息无法被解析器按原始字节语义正确处理

也就是说，本应由 XML 解析器根据文档声明处理的编码问题，被前置成了“操作系统默认编码是否刚好正确”的偶然行为。

### 3. 会导致跨平台行为不一致

同一份文档在不同机器、不同区域设置下，可能出现：

- 某些环境正常
- 某些环境报编码错误
- 某些环境虽然不报错，但解析结果不可靠

这类问题在处理 Office 文档时尤其需要避免，因为文档通常来自外部，不应该假设本机默认编码一定匹配。

## 这次修改做了什么

将 XML 文件读取方式从文本模式改为二进制模式：

```python
with open(xml_file, "rb") as f:
    xml_doc = lxml.etree.parse(f)
```

对应位置：

- `.codex/skills/docx/scripts/office/validators/base.py:763`

## 这样修改后的好处

### 1. 保留原始字节内容

使用二进制模式读取后，文件会以原始字节流形式交给 `lxml`，不会提前被 Python 用本地默认编码解码。

### 2. 让解析器尊重 XML 自身的编码声明

`lxml` 可以直接基于 XML 原始内容和声明的编码进行解析，这才是处理 XML 的正确方式。

### 3. 提高 Windows 下的稳定性

这样可以避免 Windows 本地编码对 XML 解析造成干扰，减少因为系统区域设置不同而出现的兼容性问题。

### 4. 与 XSD 读取方式保持一致

同一个方法里，XSD 文件本来就是用二进制模式读取的：

```python
with open(schema_path, "rb") as xsd_file:
```

因此把 XML 也改成二进制读取后，整体处理方式更一致，也更符合解析器对这类文件的预期。

## 结论

这次修复本质上是一个 **XML 编码处理兼容性修复**。

它解决的问题不是“业务逻辑错误”，而是“文件读取方式不够稳妥”，尤其是在 Windows 环境下，文本模式读取可能让系统默认编码错误介入 XML 解析过程。

改成二进制读取后，解析器能够基于 XML 自身声明的编码工作，行为更正确，也更稳定。
