# React / TSX 可读性守卫

## 目标
这个 skill 用来保证 React 和 TSX 代码在日常开发中更整洁、更好读、更容易维护。

它的重点不是性能优化，也不是大规模架构重构，而是让前端代码在生成和重构时看起来更清楚、更克制、更符合团队协作阅读习惯。

## 适用场景
只要任务涉及 React / TSX 代码，就优先使用这个 skill，尤其包括以下场景：

- 新建 React 组件、页面、表单、列表、弹窗、Drawer、Modal
- 修改已有 JSX / TSX 结构
- 整理组件中的条件判断、事件处理、渲染分支
- 重构已有 React 逻辑，让代码更清楚
- 在返回代码前，对生成结果做一次可读性约束
- Ant Design 场景下的 Form、Table、Modal、Drawer、Tabs、Descriptions 等常见页面结构

典型触发语句包括：
- “帮我写一个 React 表单组件”
- “整理一下这个 TSX，太乱了”
- “把这段 JSX 改得更好读”
- “优化一下这个前端组件”
- “帮我重构这个 React 页面”
- “这个 Ant Design 的弹窗表单帮我整理一下”
- “把这个列表页代码收一收，不要那么乱”

## 核心原则
当这个 skill 生效时，优先保证以下目标：

1. 代码让另一个前端同学能快速看懂。
2. 业务意图能从结构和命名中直接看出来。
3. JSX 主要负责描述结构，而不是承载密集业务逻辑。
4. 关键业务逻辑带有简短但有意义的注释。
5. 优先做局部整理，不顺手演变成大规模重构。
6. 不为了“高级感”引入过度抽象。

## 规则

### 1. 减少魔法数字，优先提常量
如果一个数字带有业务含义、布局含义、展示含义、状态含义，就优先提成有名字的常量。

推荐示例：
- `const MAX_RETRY_COUNT = 3`
- `const DEFAULT_PAGE_SIZE = 20`
- `const MODAL_FORM_COL_SPAN = 12`

只有在数字含义非常明显时，才允许直接写在代码里，例如简单的 `0`、`1`、数组下标等。

### 2. 命名必须语义化
变量名、函数名、布尔名都要直接表达业务含义，尽量避免让读代码的人再去猜。

推荐：
- `reportList`
- `selectedNode`
- `isOptionalNode`
- `handleSubmit`
- `buildSubmitParams`
- `canDeleteCurrentRow`

避免：
- `data`
- `item`
- `flag`
- `temp`
- `res`

### 3. 复杂条件先拆变量，再进入 if 或 JSX
不要把很长的条件直接塞进 `if`、三元表达式、JSX。

优先提炼出可读的中间变量，例如：
- `const isEditableRow = ...`
- `const canOpenDrawer = ...`
- `const shouldShowDeleteButton = ...`

这样后续维护时更安全，也更容易看出业务意图。

### 4. JSX 要轻，优先描述结构
JSX 应该主要描述 UI 结构，不应该承载太多判断、推导和业务拼接逻辑。

在返回 JSX 前，优先做这些整理：
- 把复杂条件提到 `return` 上面
- 把复杂文案或标签提到变量里
- 把重复渲染片段提成小块
- 避免多层嵌套三元表达式

尤其在 Ant Design 的 `Table` 列、`Form.Item` 条件渲染、`Modal` / `Drawer` 内容区域里，要优先保证扫描性。

### 5. 函数职责要单一
一个函数尽量只做一件主要事情。

如果一个函数同时在做：
- 组装参数
- 分支判断
- 调接口
- 弹提示
- 刷新列表
- 关闭弹窗

通常就应该拆一层，让流程更容易读。

但不要为了“看起来优雅”就拆得过碎。拆分的前提是：**拆完更好懂，而不是更绕。**

### 6. 代码需要简单注释
代码默认要带**简单注释**，但注释必须克制。

注释应该：
- 简短
- 解释“为什么这样做”
- 解释业务规则或不直观的约束

不要写成逐行翻译代码的废话注释。

推荐：
- 解释编辑态回填为什么优先某个字段
- 解释为什么某个字段在切换时不能清空
- 解释为什么某个按钮展示条件看起来比较特殊

避免：
- `// 设置 loading`
- `// 遍历列表`
- `// 返回组件`

### 7. 不要过度抽象
这个 skill 明确偏向“可读性优先”，而不是“抽象优先”。

默认不要：
- 为了一处逻辑新建通用 hook
- 为了少写两行代码抽一层工具函数
- 把一个简单页面拆成很多很碎的小组件
- 做与当前任务无关的结构性重构

只有当抽象能明显提升理解成本、减少重复、或者清楚表达业务边界时，才值得做。

### 8. 优先局部整理，不做顺手大改
如果用户只是要求改一个组件、表单、列表、弹窗，就优先在当前范围内做可读性优化。

不要顺手：
- 改 unrelated 文件
- 重命名大量上下游变量
- 推翻现有组织方式
- 引入新的目录结构

目标是：在最小范围内，把代码整理清楚。

## 推荐工作方式
在生成或重构 React / TSX 代码时，按下面顺序思考：

1. 先读当前组件或页面的结构。
2. 找出最影响阅读的点：
   - 魔法数字
   - 弱命名
   - 复杂条件
   - JSX 内联逻辑过多
   - 事件函数过长
   - 缺少必要注释
3. 只做当前任务真正需要的整理。
4. 尽量保持行为不变，除非用户明确要求改行为。
5. 返回代码前，按检查清单过一遍。

## 检查清单
在返回代码前，检查：

- 业务数字是否提成了常量？
- 布尔变量是否用了 `isXxx` / `canXxx` / `shouldXxx`？
- 长条件是否拆成了中间变量？
- JSX 是否以结构为主，而不是充满业务判断？
- 函数是否承担了过多职责？
- 是否加了简单但有意义的注释？
- 有没有为了抽象而抽象？
- 整理范围是否保持在当前 React / TSX 任务内？

## React / TSX 补充约束

### 优先在 return 之前整理显示状态
不建议把所有计算都写在 `return` 里面。

更推荐：
- 在 `return` 之前处理显示状态
- 提前得到 `canShowXxx`、`isXxxMode`、`submitButtonText` 这类变量
- 让 JSX 更接近“结构描述”

### 交互逻辑优先写在事件里
如果逻辑属于点击、切换、提交等用户交互，优先收敛到对应 handler 中，而不是散落在多个 effect 里。

### Ant Design 场景优先可读性
在 Ant Design 的常见场景里，要特别关注以下问题：
- `Form.Item` 条件渲染不要堆复杂表达式
- `Table` 的 `render` 和操作列不要写太长
- `Modal` / `Drawer` 内的提交逻辑不要全部堆在一个函数里
- 列表页中的筛选、分页、请求、弹窗控制要适当分层

## 边界
这个 skill 不应该自动做这些事：
- 主动发起大规模架构重构
- 无收益地新增抽象层
- 给所有代码都加很多注释
- 为了所谓整洁把小组件拆得很碎
- 顺手做性能优化，除非它自然带来更清晰的代码

## 示例

### 示例 1
**输入：** 帮我把这个 Ant Design 的弹窗表单整理一下，里面判断太多了。
**输出方向：** 把复杂判断提到 JSX 外面，命名语义化，补一两条必要注释，保持弹窗行为不变。

### 示例 2
**输入：** 生成一个 TSX 列表页，有筛选、表格和操作按钮。
**输出方向：** 使用清晰的 handler 命名，提取分页和布局常量，让列表页结构分层清楚，避免 Table 列定义太乱。

### 示例 3
**输入：** 重构这个 Drawer 里的表单提交逻辑，让代码更清楚一点。
**输出方向：** 拆开参数组装、提交动作、成功处理；补充简短注释解释关键业务意图；避免过度抽象成无意义 helper。
