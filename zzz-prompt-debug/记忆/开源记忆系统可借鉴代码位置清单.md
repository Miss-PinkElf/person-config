## 1. 结论先行

这些开源项目对当前桌宠记忆系统设计（Desktop Pet Memory System Design）的最大帮助，不是替你重新设计一套记忆系统，而是给每个设计层提供可落地参考。

当前最值得优先借鉴的是：

| 优先级 | 项目 | 最值得借鉴的层 | 结论 |
| --- | --- | --- | --- |
| P0 | `Memobase` | 用户画像（User Profile）、事件时间线（Event Timeline）、上下文组装（Context Assembly） | 最像“可产品化的主记忆层”，适合参考服务边界和召回上下文裁剪 |
| P0 | `BaiShou` | 生活史档案（Life Archive）、周期总结（Periodic Summary）、本地优先（Local-first） | 最适合参考陪伴产品里的长期生活记录和用户可控治理 |
| P0 | `Graphiti` | 记忆图谱（Memory Graph）、时间有效期（Temporal Validity）、事实失效（Fact Invalidation） | 最适合参考关系记忆和事实变化，但不要早期全量照搬 |
| P1 | `mem0` | 事实抽取（Fact Extraction）、偏好记忆（Preference Memory）、混合召回（Hybrid Retrieval） | 适合作为事实型长期记忆（Long-term Fact Memory）的工程参考 |
| P1 | `memU` | 多源写入编排（Multi-source Write Orchestration）、分类摘要（Category Summary） | 适合参考记忆服务（Memory Service）的写入 / 召回流程 |
| P1 | `langmem` | 结构化抽取（Structured Extraction）、反思执行（Reflection Execution） | 适合参考 LLM 候选记忆的 schema 化和后台反思 |
| P1 | `MemoryBank / SiliconFriend` | 遗忘强化（Forgetting / Reinforcement）、人格摘要（Personality Summary） | 适合参考陪伴感，不适合作为工程底座 |
| P2 | `LifeBook` | 日 / 周 / 月 / 年滚动压缩（Rolling Compression）和人物节点（Person Node） | 适合参考反思记忆和生活史组织，不适合直接照搬为实时服务 |
| P2 | `VCPToolBox` | 中文时间召回（Chinese Temporal Recall）、标签图谱（Tag Graph）、梦境式反思（Dream-like Reflection） | 只借鉴记忆相关插件，不要被工具生态带偏 |
| P2 | `VCPChat` | 记忆管理 UI（Memory Management UI）、图谱可视化（Graph Visualization） | 适合参考用户如何查看、编辑、删除记忆 |
| P2 | `Letta` | 核心记忆（Core Memory）、归档记忆（Archival Memory）、召回记忆（Recall Memory） | 适合参考分层抽象，不建议引入完整 Agent runtime |

最推荐的落地路线：

```text
先看 Memobase / mem0 / BaiShou
  -> 实现事件记忆（Episodic Memory）和兴趣画像（Interest Profile）
再看 Graphiti / VCPToolBox
  -> 实现轻量记忆图谱（Memory Graph）和时间召回
最后看 LifeBook / langmem / MemoryBank
  -> 实现反思记忆（Reflective Memory）和陪伴式总结
```

## 2. 按你的设计层映射

### 2.1 信号层（Signal Layer）

目标：把观察记录（Observation）、感知结果（Perception）、对话（Chat）、浏览器（Browser）、媒体（Media）、游戏会话（Game Session）统一成可追溯信号。

可借鉴：

| 项目 | 借鉴点 | 关键文件绝对路径 |
| --- | --- | --- |
| `memU` | `MemoryService` 统一组织写入流程，适合作为信号进入记忆服务的参考 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\memU\src\memu\app\memorize.py` |
| `memU` | 资源（Resource）、记忆项（MemoryItem）、分类（MemoryCategory）模型，适合参考多源证据如何落库 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\memU\src\memu\database\models.py` |
| `VCPToolBox` | 文件监听、切块、向量化、索引更新流程，适合参考“外部来源变成可检索块”的管线 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\VCPToolBox\KnowledgeBaseManager.js` |

落地建议：

- 不要把每个来源直接写成正式记忆。
- 先统一进信号记录（Signal Record），再由事件提取器（Event Extractor）生成正式记忆事件。
- 借鉴 `memU` 的服务编排，但保留你设计里的 `LLM proposes / Code validates / Store grounds with evidence` 边界。

不建议照搬：

- 不建议照搬 `VCPToolBox` 的插件链复杂度。
- 不建议一开始把所有来源都做向量化，先保证证据可追溯。

### 2.2 事件记忆（Episodic Memory）

目标：把一段时间内的多源信号融合成“发生了什么”，而不是只保存碎片文本。

| 项目 | 借鉴点 | 关键文件绝对路径 |

| `Memobase` | 上下文 API 如何把用户画像和近期事件组合进提示词 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\memobase\src\server\api\memobase_server\controllers\context.py` |
| `Graphiti` | `add_episode` 如何从片段（Episode）抽取实体和关系，并维护来源链路 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\graphiti\graphiti_core\graphiti.py` |
| `MemoryBank / SiliconFriend` | 对话历史如何转成每日事件摘要 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\MemoryBank-SiliconFriend\memory_bank\summarize_memory.py` |

落地建议：

- 你可以把 `MemoryEvent（记忆事件）` 设计成主记忆层，字段必须保留 `start_at / end_at`、`event_type`、`summary`、`confidence`、`evidence_refs`。
- `Memobase` 更适合参考“事件流 + 用户画像”的产品化服务边界。
- `Graphiti` 更适合参考 episode provenance（片段溯源）和事实演化，不必照搬图数据库。

不建议照搬：

- 不要直接把聊天 chunk 当事件。
- 不要让 LLM 输出未校验的事件直接入库。

### 2.3 兴趣画像（Interest Profile）

目标：从游戏、媒体、浏览、应用使用中沉淀“用户最近在乎什么、常回到什么”。

可借鉴：

| 项目 | 借鉴点 | 关键文件绝对路径 |

| `mem0` | 记忆项向量存储抽象，适合参考兴趣画像的可检索层 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\mem0\mem0\vector_stores\base.py` |
| `Memobase` | 用户档案（User Profile）和事件要点（Event Gist）并存，适合参考“近期兴趣 + 稳定画像” | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\memobase\src\server\api\memobase_server\models\database.py` |
| `BaiShou` | Agent 主动写入向量记忆前的去重、合并思路 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\BaiShou\lib\agent\tools\memory\memory_store_tool.dart` |

落地建议：

- 第一批兴趣画像可以只做 `frequent_game（常玩游戏）`、`favorite_artist（常听歌手）`、`favorite_streamer（常看主播）`、`recurring_media_topic（反复媒体主题）`。
- 画像不要只靠一次出现生成，至少需要频次、最近时间和置信度门槛。
- 可以参考 `mem0` 的抽取和 upsert 思路，但你的正式画像仍应由代码规则和证据链控制。

不建议照搬：

- 不建议 `mem0` 式积极 ADD-only 全量写入，桌宠场景容易误记临时情绪或隐私。
- 不建议把兴趣画像直接变成不可编辑黑盒。

| `MemoryBank / SiliconFriend` | 记忆银行如何保存历史、整体人格和元信息 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\MemoryBank-SiliconFriend\utils\memory_utils.py` |
| `MemoryBank / SiliconFriend` | 每日人格分析和长期人格摘要 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\MemoryBank-SiliconFriend\memory_bank\summarize_memory.py` |
| `BaiShou` | 会话压缩保留事实、情绪、关系动态和共同回忆的方向 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\BaiShou\lib\agent\tools\memory\memory_store_tool.dart` |

落地建议：

- 关系记忆第一版不要做太宽，先围绕 `shared_context（共同上下文）`、`followup_topic（跟进主题）`、`comfort_preference（安抚偏好）`、`sensitive_boundary（敏感边界）`。
- 关系记忆必须比普通兴趣记忆更谨慎，需要更高置信度和更强的用户可见治理。
- 可以借鉴 `LifeBook` 的“人物节点”思想，但落地时应转成你的数据库对象，而不是要求用户维护 Markdown。

不建议照搬：

- 不建议把 `LifeBook` 变成桌宠的实时存储结构。
- 不建议从单次对话推断强关系结论。

### 2.5 记忆图谱（Memory Graph）

目标：让事件、画像、兴趣、关系之间出现可解释连接，支持“我想起之前类似的事”。

可借鉴：

| 项目 | 借鉴点 | 关键文件绝对路径 |

| `Graphiti` | 实体节点（EntityNode）、片段节点（EpisodicNode）、社区节点（CommunityNode）定义 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\graphiti\graphiti_core\nodes.py` |
| `Graphiti` | 实体边（EntityEdge）的 `valid_at / invalid_at / expired_at` 时间字段 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\graphiti\graphiti_core\edges.py` |
| `VCPToolBox` | 标签共现、标签扩散和轻量主题图谱 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\VCPToolBox\TagMemoEngine.js` |
| `VCPChat` | 记忆图谱前端可视化表达 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\VCPChat\Memomodules\memo-graph.js` |

落地建议：

- 早期不要上完整图数据库（Graph Database），先用 SQLite 表实现 `memory_entities` 和 `memory_edges`。
- 第一版实体只做 `topic（主题）`、`project（项目）`、`app（应用）`、`game（游戏）`、`media_title（媒体标题）`、`interest（兴趣）`。
- 第一版关系只做 `about（关于）`、`co_occurs_with（共同出现）`、`returns_to（回到）`、`interest_of（属于兴趣）`、`derived_from（来源于）`。
- 如果需要处理“以前喜欢 A，现在不喜欢 A”，再借鉴 `Graphiti` 的事实有效期字段。

不建议照搬：

- 不建议早期引入 Neo4j / FalkorDB / Kuzu 这类完整图数据库栈。
- 不建议让自动实体抽取无限扩张，图谱脏了之后很难治理。

### 2.6 回忆服务（RecallService）

目标：根据当前上下文召回相关事件、画像、关系和图谱连接，给 `Heart（主动关心）` 与 `Agent（智能外层）` 提供“她想起了什么”。

可借鉴：

| 项目 | 借鉴点 | 关键文件绝对路径 |
| --- | --- | --- |
| `mem0` | 混合召回和实体增强打分 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\mem0\mem0\utils\scoring.py` |
| `Graphiti` | 全文、向量、图遍历、RRF、MMR 和交叉编码器重排 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\graphiti\graphiti_core\search\search.py` |
| `memU` | 先判断是否需要检索，再召回分类、记忆项、资源，并做充分性检查 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\memU\src\memu\app\retrieve.py` |
| `VCPToolBox` | 中文自然语言时间解析、时间衰减重排和 RAG 注入 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\VCPToolBox\Plugin\RAGDiaryPlugin\RAGDiaryPlugin.js` |
| `Letta` | 归档记忆搜索和 passage 管理 | `D:\Users\Mobius\Desktop\mine\AAA-code\person-config\zzz-memory-demo\letta\letta\services\passage_manager.py` |

落地建议：

- 回忆服务第一版可以不用 embedding，只基于事件类型、实体名、时间窗口、频次和置信度排序。
- 排序至少考虑 `recency（新近度）`、`frequency（频率）`、`confidence（置信度）`、`relationship_importance（关系重要性）`。
- 输出应是结构化的 `RecallResult（回忆结果）`，包含命中的事件、画像、关系、证据和适合表达的简短解释。

不建议照搬：

- 不建议一开始实现 Graphiti 的完整混合检索栈。
- 不建议把召回结果直接拼进回复，应该先经过 `Heart（主动关心）` 和 `Agent（智能外层）` 的语气 / 打扰度门禁。

- 总结必须带来源事件范围和证据引用，避免变成无来源的角色脑补。

不建议照搬：

- 不建议把反思写得太文学化，早期应优先可验证。
- 不建议从低质量事件里生成长期关系结论。

## 5. 最小落地建议

如果只做一个最小可验证版本，不要一次性实现所有层。建议只做：

1. `game_sessions（游戏会话）` 接入信号层（Signal Layer）。
2. 新增 `gaming_session（游戏会话事件）` 类型。
3. 新增 `frequent_game（常玩游戏）` 兴趣画像（Interest Profile）。
4. 新增轻量 `RecallService（回忆服务）`，能在再次打开同一游戏时找到历史事件。
5. `Heart（主动关心）` 新增 `interest_return（回到兴趣）` 触发，但必须低打扰。
6. 记忆管理页能看到这条兴趣画像，并能删除或关闭游戏采集。

这一条链路能完整证明你的设计价值：

```text
事实不是只被存下来了
  -> 它被组织成事件
  -> 事件沉淀成兴趣
  -> 兴趣可以被回忆
  -> 回忆能影响桌宠的主动陪伴表达
  -> 用户仍然可以看见和删除这条记忆
```
