# Brainstorming 指南

本文件用于 `devflow` 的对齐阶段，不只服务单一实现流程，而是服务整个长期任务主线。

## 什么时候必须先对齐

以下情况必须先做 Align：

- 需求存在歧义
- 目标不止一个实现方向
- 改动会影响多个模块、页面或阶段
- 用户明确要求先讨论、先梳理、先想清楚
- 当前任务需要写 plan 或 spec

以下情况可以使用 Mini Align，但不能完全跳过：

- 目标已经足够明确
- 改动局部且边界稳定
- 不需要完整三件套 spec

## 轻量路径与重型路径的对齐深度

### 轻量路径

轻量路径的对齐目标是确认最小必要事实：

- 目标到底是什么
- 范围内和范围外是什么
- 是否真的只需要轻量 plan 与轻量 tasks
- 有无明显风险会把任务升级成重型路径

输出应短，但不能没有互动确认。

### 重型路径

重型路径需要更完整的收敛过程：

- 明确目标、约束、成功标准
- 识别风险与未知点
- 比较 2-3 种方案及取舍
- 形成可进入 `plans/` 的方向摘要

## 对齐结束后如何沉淀到 `plans/`

对齐结果不能只留在对话里。

一旦方向基本清楚，就要把以下内容写入 `plans/`：

- 背景
- 目标
- 方案比较
- 放弃方案
- 最终选择
- 产出指向

如果是轻量路径，也至少要留下轻量 plan，而不是直接跳进实现。

## Brainstorm 与 Spec 的关系

`brainstorm` 用来理解问题与收敛方向。  
`spec` 用来表达已经正式确认的结论。

因此：

- brainstorm 先于 spec
- plan 先于重型 spec
- 不允许用 spec 代替 brainstorm 过程
- 不允许把 brainstorm 结果只留在对话中而不落盘

## 路径判断时如何解释理由

当 `devflow` 初判路径时，用一两句话说明：

- 当前判断属于轻量路径、重型路径、bug 路径还是 resume 路径
- 形成判断的主要依据
- 是否建议用户切换

示例：

```md
我判断这是轻量路径，因为目标已经明确、影响范围局部、暂时不需要完整 proposal/design/tasks。
如果你希望把讨论过程和阶段回退做得更正式，也可以切到重型路径。
```

## 禁令

- 不允许跳过 Align 直接进入 Apply
- 不允许把 brainstorm 结果只留在对话中而不落盘
- 不允许在没有 plan 的情况下，把 spec 当作过程记录替代品
