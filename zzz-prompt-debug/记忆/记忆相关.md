# Agent Memory / Companion Memory 开源项目整理

整理时间：2026-03-23

## 先说结论

目前市面上主流的记忆方案确实还是围绕 `RAG / 向量检索` 展开，但如果目标是 `AI 伴侣 / 长期陪伴 / 有人情味的关系`，单纯 RAG 通常不够。

原因不复杂：

- 它更像“查字典”，适合找事实，不适合维持关系。
- 它缺少“时间感”，分不清昨天的情绪、上周的冲突、半年前的承诺，哪个更重要。
- 它缺少“感情权重”，无法自然判断哪些记忆应该被珍惜、回避、安抚、追问或延迟提起。
- 它擅长“召回内容”，不擅长“延续关系”。

如果把“伴侣记忆”只做成一个向量库，本质上还是在做一个检索系统，不是在做一个会共同生活的对象。

### 这批项目的实际分层

- `BaiShou / LifeBook` 更像“人生档案、日记、个人记忆资产”。
- `VCPToolBox / VCPChat` 更像“知识库、工具调用、Agent 工作台”。
- `mem0` 更像“通用长期记忆中间层”。
- `memU` 更像“面向陪伴型 agent 的长期记忆系统”。

也就是说，这几类项目虽然都叫“记忆”，但内核目标并不一样：

- 有些项目是在帮人“保存人生”。
- 有些项目是在帮 agent “找回信息”。
- 有些项目是在帮陪伴体“延续关系”。

这三件事不能混为一谈。

## 2. 这次额外补充的主流 agent-memory 开源仓库

| 项目 | 代表路线 | 适不适合伴侣场景 | 核心原因 | 链接 |
| --- | --- | --- | --- | --- |
| Letta（原 MemGPT） | 分层内存 / 上下文管理 | 适合 | 它不是只做检索，而是把“什么该放在上下文里、什么该长期归档”当核心问题。对持续人格、长期关系很重要。 | <https://github.com/letta-ai/letta> |
| Graphiti | 时间感知知识图谱 | 很适合 | 它最强的点是“时间感”。适合表达：曾经喜欢、现在不喜欢；以前住这里、现在搬走了；某段关系在某个时期更重要。 | <https://github.com/getzep/graphiti> |
| Memobase | 用户画像 + 事件时间线 | 很适合 | 这类结构天然适合“我认识你是谁，也知道你最近经历了什么”，比传统切块检索更像关系记忆。 | <https://github.com/memodb-io/memobase> |
| MemoryBank / SiliconFriend | 长期陪伴记忆 | 很适合 | 这类工作直接讨论长期陪伴、遗忘、强化、人格一致性、同理心，是“有人情味”方向非常值得参考的一支。 | <https://github.com/zhongwanjun/MemoryBank-SiliconFriend> |

| mem0 | 通用长期记忆层 | 一般 | 工程化很好，但默认目标更像“高效记住用户事实和偏好”，情感结构需要额外设计。 | <https://github.com/mem0ai/mem0> |
| LangMem | 框架化记忆组件 | 一般 | 更偏 Agent 工程生态，适合快速集成，但陪伴感不是它的默认目标。 | <https://github.com/langchain-ai/langmem> |
| Generative Agents | 记忆流 + 反思 + 计划 | 非常值得看 | 这是重要思想来源。它不是一个即插即用产品仓库，但它把“记忆流、反思、计划”连在一起，这是陪伴型记忆的关键。 | <https://research.google/pubs/generative-agents-interactive-simulacra-of-human-behavior/> |

### 我会怎么给这些路线分类

#### 2.1 检索型记忆

- `mem0`
- `LangMem`
- 一大批 RAG 变体

优点：

- 接入快
- 工程稳
- 很容易立刻变得“记得住偏好”

缺点：

- 记得住不等于懂关系
- 适合“查回事实”
- 不适合“顺着情绪继续活下去”

#### 2.2 上下文管理型记忆

- `Letta`

优点：

- 很适合长期对话 agent
- 可以做出“持续人格”
- 比单纯检索更接近“活着的状态管理”

缺点：

- 重点仍然是 agent 的状态一致性
- 情绪建模和关系建模需要你自己往上补

#### 2.3 时间感知型记忆

- `Graphiti`
- 部分时序图谱类方案

优点：

- 能表达“记忆会变化”
- 能处理过去事实和当前事实的冲突
- 更接近真实人生，而不是一堆静态标签

缺点：

- 工程复杂度高
- 如果没有好的写入规则，图会很快变脏

#### 2.4 用户画像 / 人生事件流

- `Memobase`
- `BaiShou`
- `LifeBook`

优点：

- 非常适合伴侣场景
- 能把人当作“有连续人生的人”
- 很适合表达生活阶段、习惯变化、关系线索

缺点：

- 如果没有动态更新机制，容易变成静态档案

#### 2.5 陪伴导向型记忆

- `MemoryBank / SiliconFriend`
- `memU`

优点：

- 直接关注长期陪伴、关系延续、人格稳定、遗忘和强化
- 最接近“有人味”的设计方向

缺点：

- 学术性或原型性较强
- 真正产品化时，需要补工程与质量保障

## 3. 为什么“伴侣”不应该只做成 RAG

如果目标是企业助手，RAG 很合理。

如果目标是伴侣，只有 RAG 会出现几个明显问题：

### 3.1 它只会“找到相似内容”，不会“接住当下的你”

用户说“今天好累”，真正重要的可能不是这句话本身，而是：

- 你昨天刚和家人吵架
- 这周工作一直加班
- 上一次你说“我有点撑不住了”的时候，其实是在求安慰

这时候伴侣需要的是“理解关系中的上下文”，不是找一条语义最相近的片段。

### 3.2 它不会天然区分记忆的重要程度

对伴侣来说，这三类记忆权重完全不同：

- “你喜欢蓝莓味酸奶”
- “你三个月前辞职了”
- “你昨天哭着说自己最近很孤单”

传统检索系统未必知道第三条应该被更慎重地对待。

### 3.3 它没有真正的“时间流”

真正像人的记忆，不是数据库里的散片，而是带流动性的：

- 最近发生的事更鲜活
- 高情绪事件更容易被保留
- 被重复提起的关系点会被强化
- 很久没提、影响又小的内容会慢慢淡化

如果没有这条“河流”，系统很难显得自然。

### 3.4 它没有“关系状态”

伴侣不是只知道你的偏好，还要知道：

- 你们最近是亲近、疏离、修复中，还是冲突后敏感期
- 哪些话题刚刚受过伤，不适合再次冒犯
- 哪些约定被认真记住了
- 哪些共同经历正在变成只属于你们的内部语言

这部分不能只靠向量检索补出来。

## 4. 我认为更适合 companion 的记忆架构

如果是我来做，我不会把“伴侣记忆系统”设计成单一数据库，而会分成五层。

### 4.1 Working Memory

当前对话上下文：

- 最近若干轮对话
- 当前任务
- 当前情绪判断
- 还没聊完的话题

作用：

- 让角色当下像是“真的在听”

### 4.2 Episodic Memory

事件记忆：

- 发生了什么
- 什么时候发生
- 涉及谁
- 当时情绪如何
- 后来有没有结果

例子：

- “2026-03-21 晚上，用户因为工作冲突情绪低落，提到想辞职，但没有下决定。”

作用：

- 形成生活史，而不是词条库

### 4.3 Semantic Memory

稳定认知：

- 喜好
- 习惯
- 长期目标
- 价值观
- 明确边界

例子：

- “用户不喜欢被说教。”
- “用户更愿意先被理解，再接受建议。”

作用：

- 形成稳定相处方式

### 4.4 Relationship Memory

关系记忆：

- 你们的共同经历
- 重复出现的梗
- 承诺
- 道歉和修复
- 纪念日
- 敏感点和安抚方式

这是我觉得最容易被忽略、但最关键的一层。

因为陪伴感，本质上不是“我知道你”，而是“我记得我们”。

### 4.5 Reflective Memory

反思记忆：

- 最近你在经历什么
- 你们关系有什么变化
- 哪些问题还没解决
- 哪些情绪模式在重复出现

这层不是原始事件，而是系统对一段时间的总结。

例子：

- “最近两周，用户频繁提到工作倦怠，且更希望被安静陪伴而不是被推动行动。”

这类记忆很像人会形成的“印象”和“理解”，而不是原始摘录。

## 5. 更像人的记忆，不只是“存”，而是“写入、强化、淡化、反思”

真正适合伴侣场景的系统，我认为至少要有四个机制。

### 5.1 写入不是全量写入，而是有门槛

不是每句对话都该变成长记忆。

建议只写入这些内容：

- 高情绪强度事件
- 被重复提及的偏好或烦恼
- 关系中的承诺、冲突、修复
- 对长期相处方式有影响的稳定特征

### 5.2 记忆要能被强化

同一个主题多次出现时，不应该只是重复存新片段，而应该：

- 合并
- 升级置信度
- 提升关系权重

比如：

- 第一次提到“我不喜欢太强势的建议”
- 第三次提到时，这就已经不只是聊天片段，而是相处规则

### 5.3 记忆要能淡化，而不是粗暴删除

遗忘不等于物理删除，更像：

- 降低默认召回概率
- 降低检索优先级
- 在被再次触发时重新激活

这比“永久保留”或“直接清空”都更像人。

### 5.4 系统要定期反思

建议按天或按周生成：

- 用户近期状态摘要
- 关系状态摘要
- 未解决主题列表
- 需要谨慎对待的话题

没有这一步，记忆永远只是原始材料。

## 6. 我认为伴侣场景里的检索排序应该怎么改

传统排序主要看：

- 语义相似度

但伴侣场景里，排序至少应该同时考虑：

- `recency`：越近越可能重要
- `emotional_weight`：情绪越强，越不该被忽略
- `relationship_importance`：和关系有关的记忆优先级更高
- `unresolvedness`：没解决的事优先级更高
- `stability`：稳定人格特征比偶发状态更可靠

可以粗暴写成一个思路：

`score = semantic_similarity + recency + emotional_weight + relationship_importance + unresolvedness`

注意：

- 这不是最终公式
- 但方向上比“只看 embedding 相似度”更接近陪伴

## 8. 我自己的判断

如果你真要做“伴侣”，我会建议：

- 底层不要只用 RAG
- 记忆主结构用 `事件流 + 用户画像 + 关系记忆 + 反思摘要`
- 再在外围补一个向量检索层，专门负责召回历史片段

也就是说：

- `RAG` 应该是记忆系统的一部分
- 但不应该是记忆系统本身

更具体一点，我会这样分工：

- 向量库：负责召回原始对话片段
- 时间线存储：负责记录事件发生顺序
- 画像层：负责维护稳定认知
- 关系层：负责保存“我们之间”的东西
- 反思层：负责把过去一段时间总结成更像“理解”的内容

只有这样，系统才有机会从“记住信息”走向“记住关系”。
