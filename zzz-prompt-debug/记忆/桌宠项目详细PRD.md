# 桌宠项目详细 PRD

## 1. Executive Summary

- **Problem Statement**: 现有 AI 助手大多以“工具调用”和“问答效率”为核心，缺少持续陪伴感、人格连续性和对用户生活习惯的长期理解，无法满足“像一个人一样陪伴”的目标。该项目需要在毕业设计可交付的时间约束下，先完成一个可运行、可演示、可扩展的桌宠系统，再逐步演进为具备长期关系记忆的陪伴式 AI 产品。
- **Proposed Solution**: 构建一个基于 `Electron + React + Live2D + Python Agent Backend` 的桌面 AI 桌宠，前端提供透明悬浮宠物、拖拽交互、对话入口、状态反馈和基础行为控制，后端提供 LLM 对话、分层记忆、用户状态感知和可插拔能力模块。产品采用“毕业设计 MVP -> 演示增强版 -> 长期产品雏形”三阶段路线，在前期优先保障可交付和答辩通过。
- **Success Criteria**:
  - 毕设 MVP 版本可在 `Windows` 环境稳定运行，完成安装、启动、模型展示、聊天、基础记忆写入与读取的完整闭环，连续运行 `30` 分钟无崩溃。
  - 用户在 `3` 步以内可完成首次对话，桌宠主界面启动时间不超过 `5` 秒，单轮文本对话首响应时间在本地网络正常条件下不超过 `6` 秒。
  - 在 `20` 条预设记忆测试样本中，系统对“用户偏好/习惯/近期事件”的召回准确率达到 `>= 80%`，且错误召回中不得包含用户明确关闭采集的隐私信息。
  - 可演示版支持不少于 `5` 个稳定场景演示：拖拽与贴边、悬浮菜单、对话气泡、记忆召回、基于当前窗口的轻量上下文反馈。
  - v2 产品雏形中，记忆模块、感知模块、模型控制模块和对话编排模块均为可替换组件，任一模块替换时不要求前端主交互层重构。

## 2. User Experience & Functionality

### User Personas

- **Persona A: 毕业设计作者本人**
  - 目标：先完成一个可以提交、可以答辩演示、可持续迭代的桌宠系统。
  - 核心诉求：功能闭环、架构清晰、展示效果明显、技术路线可解释。
- **Persona B: 日常陪伴型重度用户**
  - 目标：桌宠不是工具入口，而是一个长期陪伴、会记住自己习惯和情绪线索的数字生命体。
  - 核心诉求：自然互动、低打扰、有人格感、能逐渐“更懂我”。
- **Persona C: 答辩老师/演示观察者**
  - 目标：快速理解系统价值、核心创新点和工程落地能力。
  - 核心诉求：演示稳定、功能边界清晰、隐私设计合理、技术方案自洽。

### User Stories

#### Story 1: 桌宠可被看见、拖动并持续悬浮

- As a 用户, I want to 在桌面上看到一个透明悬浮的 Live2D 桌宠并能自由拖拽, so that 它像真实桌宠一样存在于我的桌面环境中。

#### Acceptance Criteria

- Electron 主窗口必须支持透明背景、无边框、置顶显示。
- Live2D 模型必须支持 `.model3.json` 格式并可正常加载、居中显示、缩放适配。
- 用户可通过鼠标拖拽移动桌宠位置，释放后位置被保留到本地配置中。
- 鼠标悬浮在桌宠上时，必须显示与视觉稿一致的右侧竖向 toolbar，并包含设置入口、缩放入口及后续扩展按钮位。
- 桌宠外层必须提供悬浮高亮边框，鼠标移出后保留短暂延迟再消失，避免交互闪烁。
- 用户必须可以通过边框拖拽移动窗口，并通过右下角缩放手柄同时调整 Electron 窗口尺寸与 Live2D 模型显示比例。
- 当窗口尺寸变化时，外框、模型、toolbar、拖拽把手和缩放手柄必须表现为同一套联动缩放系统，而不是固定像素叠加层。
- 桌宠窗口必须同时具备最小值和最大值限制，避免缩到不可用或放大到失衡。
- 当窗口尺寸变化或模型切换时，模型不得超出可视区域。
- 模型与边框之间的安全留白应尽量收紧，默认状态下模型需要更贴近边框主体区域。

#### Story 2: 用户可以随时与桌宠对话

- As a 用户, I want to 点击桌宠或快捷方式打开对话入口并发送消息, so that 我可以把它当成一个常驻陪伴对象而不是单次工具。

#### Acceptance Criteria

- 系统必须提供常驻小对话框或点击唤起输入框。
- 用户发送文本消息后，系统必须展示桌宠气泡回复和对话面板记录。
- 对话记录至少保留最近 `20` 轮本地会话内容，重启后可恢复最近会话摘要。
- 在网络或模型不可用时，系统必须给出明确降级提示，而不是无响应。
- 前端消息状态至少区分 `发送中`、`成功`、`失败` 三种状态。

#### Story 3: 桌宠会逐渐记住用户

- As a 用户, I want to 桌宠记住我的偏好、习惯和近期经历, so that 它在后续对话里表现得更像“认识我”的个体。

#### Acceptance Criteria

- 后端必须支持独立的记忆模块，至少区分 `工作记忆`、`事件记忆`、`语义记忆` 三类。
- 系统必须从对话中抽取可持久化信息，并写入统一记忆层。
- 用户下一次对话时，系统必须能召回与当前话题相关的记忆片段，并在回复中体现。
- 用户必须可以查看、删除或关闭某类记忆采集。
- 记忆模块与主对话编排解耦，替换 `mem0` 或其他记忆实现时，不影响前端交互层。

#### Story 4: 桌宠会感知“我现在在做什么”，但不越界

- As a 用户, I want to 桌宠基于当前活动窗口或使用习惯给出轻量反馈, so that 它看起来更懂我的当下处境，但不会显得冒犯。

#### Acceptance Criteria

- MVP 阶段仅允许采集当前活动窗口名称、应用名、使用时段等低风险上下文。
- 所有系统感知项必须提供开关，并默认关闭高敏感能力。
- 当系统基于上下文做出提示时，必须可追溯到明确来源，例如“检测到你正在使用 VSCode”。
- 未经用户主动开启，系统不得读取摄像头、麦克风、浏览器历史或屏幕内容。
- 若感知模块异常或权限不足，主对话能力仍必须可独立运行。

#### Story 5: 桌宠在答辩场景中可清晰展示技术亮点

- As a 毕设作者, I want to 在有限时间内演示桌宠从展示、对话到记忆召回的完整流程, so that 评审可以快速理解项目价值和技术难点。

#### Acceptance Criteria

- 系统必须具备一套 `5-8` 分钟内可稳定复现的标准演示脚本。
- 演示脚本至少覆盖模型渲染、拖拽、悬浮菜单、对话、记忆召回、上下文感知中的 `5` 个能力点。
- 答辩模式下必须支持一键清理测试数据或切换到演示数据集。
- 出现模型接口失败、记忆为空、能力关闭等情况时，仍需存在兜底演示路径。

### Non-Goals

- 本阶段不追求完整实现“完全像人”的情感智能，只实现可验证的陪伴感基础能力。
- 毕设 MVP 不做无授权自动执行系统操作，不做高风险“替用户自主决策”的 Agent 行为。
- 毕设 MVP 不默认开启摄像头、麦克风、屏幕分析、浏览器详细历史读取等高隐私采集。
- 本阶段不追求跨平台完全一致，优先保证 `Windows` 版本稳定可交付，`macOS` 作为兼容性预研或次优支持。
- 本阶段不自研底层 TTS、ASR、表情识别大模型，如需接入，优先采用可替换的现成服务或开源组件。

## 3. AI System Requirements

### Tool Requirements

- **LLM 编排层**
  - 提供多轮对话能力、系统提示词管理、用户状态注入、记忆召回结果融合。
  - 后端语言为 `Python`，框架可选 `LangChain` 或轻量自定义编排；当前具体实现标记为 `TBD`，但必须支持后续替换。
- **记忆系统**
  - 推荐以独立模块封装，优先验证 `mem0` 作为 MVP 长期记忆中间层。
  - 为长期产品演进预留 `episodic / semantic / relationship` 分层结构。
  - 存储层可选本地文件、SQLite、向量数据库或混合方案；MVP 以低部署复杂度为优先。
- **感知与上下文采集**
  - MVP：活动窗口名、应用名、使用时段、前台时长统计。
  - 后续扩展：浏览器标签分析、屏幕理解、音频事件、表情识别、多模态推断。
  - 所有采集模块必须独立开关，可插拔接入。
- **桌宠行为控制**
  - 前端需支持 Live2D 模型加载、动作切换、表情切换、位置移动和基础状态映射。
  - 需兼容当前已验证组合：`pixi.js@6` + `pixi-live2d-display@0.4.0`。
- **通信与集成**
  - Electron 与 React 通过前端状态层和 IPC 通信。
  - Electron 主进程与 Python 后端通过本地 HTTP、WebSocket 或子进程桥接通信；最终实现标记为 `TBD`，但必须满足本地开发和一键启动要求。

### Evaluation Strategy

- **功能验收**
  - 按用户故事逐条验证 AC，形成毕设演示 checklist。
  - 至少准备 `15` 条功能测试用例，覆盖启动、模型加载、拖拽、聊天、记忆写入、记忆召回、异常提示。
- **记忆质量评估**
  - 构建不少于 `20` 条测试样本，覆盖用户偏好、习惯、近期事件和禁记内容。
  - 指标包括 `记忆写入成功率`、`召回准确率`、`错误召回率`、`隐私违规召回次数`。
- **陪伴感评估**
  - 用 `10` 个典型对话场景评估是否具备“像在和熟悉对象说话”的基本体验。
  - 采用人工标注维度：`自然度`、`连续性`、`记住我的程度`、`不打扰感`，每项 `1-5` 分，目标均值 `>= 3.5`。
- **系统稳定性评估**
  - 本地连续运行 `30` 分钟不崩溃。
  - 连续进行 `20` 轮文本对话时，前端无严重卡顿或内存异常增长。
- **答辩可演示性评估**
  - 演示流程在 `3` 次完整彩排中成功率达到 `100%`。
  - 任一能力失效时必须有明确降级说明和替代演示路径。

## 4. Technical Specifications

### Architecture Overview

- **总体架构**
  - 采用前后端同仓结构，根目录保留统一 `package.json` 负责启动入口。
  - 目录原则为：
    - `frontend/`: Electron + React + Vite + TypeScript + Ant Design + Live2D 交互层。
    - `backend/`: Python Agent 服务、记忆模块、感知模块、模型调用与编排逻辑。
- **前端层**
  - Electron 负责透明桌面窗口、置顶、窗口行为和本地能力桥接。
  - React 负责状态管理、悬浮菜单、对话框、设置面板和桌宠交互 UI。
  - React 层需额外维护桌宠 hover 态、toolbar 显隐、边框延迟消失、窗口拖拽与右下角缩放手柄的交互状态。
  - 桌宠容器层需维护统一布局 token：同一套缩放基准同时驱动边框圆角、toolbar 尺寸、拖拽把手、缩放手柄和 Live2D 模型可视区 inset。
  - 渲染层必须监听真实窗口 resize，并同步最新 bounds，保证不是只有自定义缩放手柄才能触发模型与 UI 重算。
  - 样式采用 `CSS Modules + :global` 的约定方式，保持模块隔离与局部覆盖。
  - `frontend/models` 存放模型资源和必要运行时文件。
- **桌宠渲染层**
  - 使用 `pixi.js@6` 和 `pixi-live2d-display/cubism4` 实现 `.moc3 / .model3.json` 模型加载。
  - 必须注册 `Live2DModel.registerTicker(PIXI.Ticker)` 并处理模型居中、缩放和生命周期销毁。
- **后端能力层**
  - 对话编排器：组织系统提示词、用户输入、记忆召回和上下文感知结果。
  - 记忆服务：负责抽取、存储、检索、删除和关闭策略。
  - 感知服务：负责活动窗口、应用使用信息等低风险上下文采集。
  - 配置服务：负责模型供应商配置、权限开关、演示模式和日志级别。
- **数据流**
  - 用户输入 -> React 对话组件 -> Electron/IPC -> Python 对话服务。
  - Python 服务 -> 读取工作记忆和长期记忆 -> 调用 LLM -> 返回回复与状态标签。
  - 前端根据状态标签触发表情、动作、气泡文本和日志展示。
  - 感知服务周期性生成上下文摘要，仅在权限开启时注入对话编排层。

### Integration Points

- **前端依赖**
  - `Electron`
  - `React`
  - `Vite`
  - `TypeScript`
  - `Ant Design`
  - `sass`
  - `pixi.js@6`
  - `pixi-live2d-display@0.4.0`
- **后端依赖**
  - `Python` 虚拟环境，必须提供标准启动脚本。
  - LLM 编排可选 `LangChain` 或轻量方案，最终选型 `TBD`。
  - 记忆方案优先验证 `mem0`，长期可评估 `Letta / Graphiti / memU` 等方向。
- **进程通信**
  - 根目录统一启动脚本负责同时启动前端和后端。
  - 通信协议需满足本地开发调试方便、日志可见、错误可追踪三项要求。
- **模型与资源**
  - Live2D Core 文件需本地引入。
  - 模型资源目录必须适配 Vite 静态资源规则，避免运行时路径错误。
- **平台集成**
  - MVP 重点验证 `Windows`。
  - 若接入系统活动窗口识别，需要明确使用的 Win API 或跨平台替代层；当前 `macOS` 平台相关能力标记为 `TBD`。
- **认证与账号**
  - 毕设版本默认单用户本地运行，不引入远程账号体系。
  - 若接入云端模型 API，则采用本地配置密钥文件或环境变量方式管理。

### Security & Privacy

- 所有高敏感采集能力默认关闭，采用显式授权开启机制。
- 设置页必须明确标识每种采集能力的数据来源、用途、存储位置和关闭方式。
- 用户必须可以删除本地记忆、清空演示数据并关闭记忆写入。
- MVP 不在后台静默上传用户原始行为数据。
- 对话日志、记忆内容、上下文采样记录必须支持本地可见和本地删除。
- 若启用第三方云模型，需在设置中提示数据可能被发送到外部服务。
- 自主执行类能力在毕设阶段默认不开放，避免安全和伦理边界失控。

## 5. Risks & Roadmap

### Phased Rollout

#### MVP: 毕业设计可交付版本

- 目标：保证可运行、可演示、可答辩、结构清晰。
- 范围：
  - 透明 Electron 窗口与 Live2D 模型展示。
  - 模型拖拽、悬浮菜单、基础设置。
  - 文本对话与气泡回复。
  - 基础记忆写入与读取闭环。
  - 低风险上下文感知：活动窗口名/应用名/使用时段。
  - 本地配置、启动脚本、演示模式。
- 不纳入：
  - 摄像头、麦克风、屏幕理解。
  - 无授权自动执行任务。
  - 复杂情绪识别与高自治行为。

#### v1.1: 可演示 + 可扩展架构版本

- 目标：增强演示效果，补足模块边界，提升“像一个人”的体验。
- 范围：
  - 更自然的表情和动作联动。
  - 记忆分层优化，加入事件摘要与关系标签。
  - 更完善的设置页和权限开关。
  - 更稳定的上下文感知和规则触发。
  - 可切换模型提供商与记忆实现。

#### v2.0: 长期维护的正式产品雏形

- 目标：从毕设作品演进为具备长期陪伴潜力的产品原型。
- 范围：
  - 引入关系记忆和时间感知机制。
  - 增强长期人格连续性和对习惯变化的理解。
  - 评估语音、情绪、多模态输入能力。
  - 探索更自然的任务协助，但保留强授权边界。

### Technical Risks

- **风险 1: “像一个人”目标过于抽象，容易失控**
  - 说明：如果不把“陪伴感”拆解成可评估指标，项目会陷入无限扩展。
  - 应对：将目标降解为自然度、连续性、记忆召回、低打扰感四类指标，MVP 只做基础达标。
- **风险 2: Live2D 与 Electron 技术兼容性不稳定**
  - 说明：渲染链路和依赖版本容易导致模型不显示、模糊或重复初始化。
  - 应对：固定已验证版本组合，沿用 `pixi.js@6 + pixi-live2d-display@0.4.0`，避免随意升级。
- **风险 3: 记忆系统耦合过深，后续难以替换**
  - 说明：若直接把记忆逻辑写死在对话流程中，后续切换 `mem0`、图谱或文件系统式记忆成本高。
  - 应对：抽象统一 Memory Service 接口，隔离写入、召回、删除和权限策略。
- **风险 4: 高隐私能力引发伦理与答辩质疑**
  - 说明：摄像头、麦克风、浏览器历史和自动执行会显著提升风险。
  - 应对：MVP 默认关闭并排除出核心交付范围，仅作为未来研究方向说明。
- **风险 5: Python 后端环境和依赖管理复杂**
  - 说明：毕设环境切换、答辩机器部署和虚拟环境问题可能直接影响演示。
  - 应对：提供明确虚拟环境初始化脚本、依赖锁定文件和一键启动入口。
- **风险 6: 多平台兼容导致时间被摊薄**
  - 说明：同时追求 Win/mac 双平台稳定会显著增加工作量。
  - 应对：优先以 `Windows` 为主平台达标，`macOS` 只保持基础可运行或列为次级目标。
