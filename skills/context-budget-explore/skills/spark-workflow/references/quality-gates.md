# 质量门禁

在 SPARK-workflow 的 Apply 和 Close 阶段使用本文件。

## 实现门禁

- 修改代码前先读取当前 artifact
- 严格按 `tasks.md` 实现，不要凭记忆实现
- 明确标记任务状态
- 如果任务意图不清，先问清楚，不要猜
- 如果暴露设计缺陷，先停止并更新 artifact

## 调试门禁

出现以下情况时，触发 systematic debugging：

- 测试出现非预期失败
- UI 行为与设计不一致
- 某次修复尝试失败
- 根因未知

升级规则：

- 三次修复尝试失败后，不要继续打补丁，应转而质疑设计或架构

## 验证门禁

在声称完成之前：

- 重新读取 `proposal.md`
- 重新读取 `tasks.md`
- 运行相关验证命令
- 检查运行时行为
- 在适用时确认边界状态和控制台健康状况

## 前端卫生门禁

除非项目已有另一套明确模式，否则优先遵循以下默认约束：

- 类型放在独立类型文件或类型区块
- 常量放在独立常量文件
- mock 数据与生产逻辑分离
- 优先使用模块化样式组织
- 避免 `!important`，也避免没有明确理由的临时内联样式

## 审查门禁

- 在有意义的里程碑后请求 review
- 从技术层面评估反馈
- 如果反馈意味着需求或设计变化，回退到 Propose
