# 经验沉淀

- 在评估外部插件是否能承接现有工具能力前，先确认本地运行时、认证和入口脚本都可用，可以避免后续结论建立在错误前提上。
- 对这类“替代内置工具”的探索，优先把问题拆成“工具层替换”与“工作流层代理”两层；前者通常受宿主限制，后者更容易落地。
- 当目标是让外部模型承担搜索/抓取时，最好把它限制为“结构化采集器”，不要同时让它负责理解与总结，否则输出契约不稳定。
- 如果准备做代理 skill，先稳定 JSON 协议与失败格式，再考虑触发描述优化和总路由封装，能减少返工。
- 对结构化代理 skill，最好提前约定空结果也返回合法 JSON：搜索返回空 `results` 数组，抓取失败则直接走 `error`，不要混入“未找到/抓取失败”的解释文字。
- 若后续还要接 Claude 侧二次处理，最好明确要求“只输出 JSON、不要代码块围栏、不要前后缀说明”，否则模型层很容易发生轻微格式漂移。
- 真实试跑表明：即使 Codex 能大体遵守纯 JSON 返回，也可能出现协议层漂移，例如无结果时省略 `query`，或失败时扩展成嵌套 `error` 对象并额外带 `warnings`。
- 如果当前阶段目标是验证旁路能力是否跑得通，可以先用“是否有可消费返回”作为验收标准，不必一开始就把字段 shape 卡得过死；等路径跑通后再逐步收紧契约，推进效率更高。
- 对 Codex rescue 真实试跑时，偶尔会出现主结果未直接展示、需要二次索取 stdout 的情况；这更适合先做可用性判断，而不是在第一轮就做严格自动校验。
- 真实试跑还暴露出一个更实际的问题：即使本地 setup 曾显示 ready，具体调用链路在某些分支上仍可能因为远端令牌失效而出现 `401 Unauthorized`。因此“本地 ready”不等于“端到端可用”，仍需要至少一次真实成功请求验证。
- 复测表明 401 不是两个 proxy skill 文案本身必然导致的问题：同一套 skill 再次调用时，search 与 fetch 的最小成功用例都能返回结构化结果。更可能是底层 Codex 运行时、远端网关或会话令牌的偶发状态问题。
- 如果当前目标是先跑通，可接受底层返回的宽松 schema，例如 `webfetch_result` 中出现 `content`、`links` 这类变体；等稳定后再做统一字段映射。
